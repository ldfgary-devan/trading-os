# 一、项目认知

## 项目身份

TradingOS 是一个 **AI 交易能力训练系统**。

它**不是**选股软件、自动交易软件或市场预测软件。

它的唯一目标是：**帮助交易者持续提升交易能力。**

---

开始任何任务之前，必须按顺序阅读以下文档：

1. `docs/CONSTITUTION.md` — 项目价值观与最高原则
2. `README.md` — 项目定位与边界
3. `docs/CAPABILITY_MAP.md` — 五层交易能力模型
4. `docs/DATA_MODEL.md` — 交易生命周期数据模型
5. `docs/PROJECT_RULES.md` — 项目开发规则

如果任务涉及业务逻辑，**禁止跳过以上任何一份文档。**

---

# 三、任务规划

## 判断任务类型

- 纯文档修改 → 直接执行
- 单文件代码修改 → 确认上下文后执行
- 多文件变更 → 先列出影响范围，再执行
- 架构决策 → 先讨论，再实施

## 每个功能必须回答

1. 它训练的是哪一种交易能力？（Perception / Thinking / Execution / Reflection / Evolution）
2. 它帮助交易者解决什么真实问题？

如果无法回答，**暂停并提醒用户**。

---

# 四、协作规范

## 何时主动提问

- 需求模糊，存在多种合理理解
- 涉及架构决策
- 需要用户确认设计方向

## 何时直接执行

- 用户已给出明确、具体的指令
- 改动范围清晰且无歧义
- 属于已讨论过的方案的实施步骤

## 不做什么

- 不为了"看起来更好"而引入未讨论的功能
- 不在没有真实数据支持的情况下添加分析逻辑
- 不预测市场、不推荐股票、不替代用户做交易决策
- 不跳过 Constitution 和 PROJECT_RULES 的约束

---

# 五、数据分层意识

在涉及 Trade 数据时，必须保持三层分离：

1. **事实数据**（原始数据，不可变）
2. **分析数据**（基于规则计算，必须有明确规则）
3. **教练数据**（AI 生成，必须可解释）

三层不得混在同一结构或同一逻辑中。

---

# 六、能力映射意识

任何分析、功能或模块，最终必须回归到五层能力循环：

```
Perception → Thinking → Execution → Reflection → Evolution
```

如果一个分析不能告诉用户"哪一项能力需要提升"，则它是不完整的。

---

# 七、开发节奏

- 小步迭代，不追求一次性完成
- 每次改动只解决一个问题
- 完成一个 Sprint 目标后，停下来同步状态
- 不在 Sprint 进行中引入无关变更
